ユーザーの要望をヒアリングし、タスクに分割して `docs/06-tasks.md` に追記する。

**入力**: $ARGUMENTS（やりたいことの概要）

**Steps**

1. **入力の確認**

   引数が空の場合は **AskUserQuestion tool** で聞く:
   > "何をやりたいですか？ やりたいことを教えてください。"

2. **プロジェクトのコンテキストを把握する**

   以下のファイルが存在すれば読み、プロジェクトの現状を把握する:
   - `CLAUDE.md`
   - `docs/01-concept.md`
   - `docs/02-service-design.md`
   - `docs/03-ui-design.md`
   - `docs/04-system-design.md`
   - `docs/05-development.md`
   - `docs/06-tasks.md`

3. **大筋の方針を検討し、不明点をユーザーに質問する**

   入力内容とプロジェクトのコンテキストを踏まえて、実現方針のドラフトを考える。
   その上で、タスク分割に必要な判断材料が不足している場合は **AskUserQuestion tool** で質問する。

   質問すべき観点の例:
   - 技術的なアプローチの選択肢がある場合、どちらを採用するか
   - スコープが曖昧な場合、どこまでやるか
   - 既存機能との統合方針が不明な場合
   - 優先度やリリース時期に関する制約

   **重要**:
   - 一度に大量の質問をせず、最も重要な不明点に絞って質問する（最大3問程度）
   - 明確に判断できることは自分で決めて進める
   - 質問が不要な場合はスキップしてよい

4. **タスクに分割する**

   方針が固まったら、実装可能な粒度のタスクに分割する。

   タスク分割のガイドライン:
   - **目的ベースのPBI（プロダクトバックログアイテム）のような粒度で切る**。実装手順や技術的な作業単位ではなく、「ユーザーや開発者にとってどんな価値を届けるか」という目的を軸にタスクを定義する
   - 各タスクは独立して着手・完了できる粒度にする
   - 依存関係がある場合は順序を考慮して並べる
   - 各タスクには概要と、関連するドキュメントやファイルへの参照を含める

   フォーマット:
   ```markdown
   - [ ] タスク名
     - 概要の説明
     - @関連ファイルやドキュメントへの参照
   ```

5. **タスクをバックログに追記する**

   `docs/06-tasks.md` の「## 未着手」セクションの末尾（`---` の直前）に、分割したタスクを追記する。

   既存のタスクがある場合はその後に追加する。既存タスクを変更・削除しないこと。

6. **結果を報告する**

   追記したタスクの一覧を表示し、ユーザーに確認を促す:
   - 追加したタスク数と概要
   - 「内容を確認して、修正が必要であれば教えてください」
